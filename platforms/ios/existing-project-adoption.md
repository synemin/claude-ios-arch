# Existing Project Adoption

Use this guidance when the plugin is introduced into a project that is already under development.

## First Questions
- What stage is the project in?
- What is already working that should not be broken?
- What pain is motivating adoption now?
- What deadlines constrain change?

## Preserve First
Prefer preserving:
- working release flow
- known-good architecture boundaries
- stable team habits that are not harmful
- existing tools that solve real pain

## Change First
Prioritize:
- structural risks blocking feature delivery
- hidden debt causing repeated bugs
- missing observability
- environment drift that hurts reproducibility
- unclear ownership boundaries

## Avoid
- broad rewrites without evidence
- forcing greenfield patterns onto a live project
- replacing tools just for stylistic purity
